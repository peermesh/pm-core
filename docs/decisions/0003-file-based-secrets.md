# ADR-0003: File-Based Secrets Management

## Context

Secrets (database passwords, API keys, signing keys) must be injected into containers securely. Common approaches include:

- Environment variables in `.env` files
- Docker Swarm secrets
- External secret managers (HashiCorp Vault, Doppler)
- File-based injection via Docker Compose `secrets` directive

The solution must:

- Work with Docker Compose (not Swarm mode)
- Not expose secrets in `docker inspect` output
- Not require external cloud services
- Fit commodity VPS resource constraints
- Require zero daily maintenance

---

## Decision

**We will use file-based secrets** stored in a `./secrets/` directory with 700/600 permissions, mounted via Docker Compose's `secrets` directive to `/run/secrets/` in containers.

Secrets are:
- Generated by an idempotent bootstrap script (`scripts/generate-secrets.sh`)
- Never stored in environment variables with actual values
- Mounted read-only into containers
- Injected via `_FILE` suffix convention for compatible applications

For version-controlled encrypted storage (optional), SOPS with Age encryption is recommended.

---

## Alternatives Considered

### Option A: Environment Variables Direct

**Description**: Store secrets in `.env` files loaded by Docker Compose, passed as environment variables.

**Pros**:
- Simplest approach
- Universal application support

**Cons**:
- Exposed in `docker inspect` output
- Visible in `/proc/[pid]/environ` inside containers
- Can leak to logs if application logs environment

**Why not chosen**: This is a fundamental security anti-pattern. Secrets visible in `docker inspect` would fail any professional security audit.

### Option B: HashiCorp Vault or Similar

**Description**: Deploy a dedicated secret manager service.

**Pros**:
- Enterprise-grade secret management
- Dynamic secret rotation
- Audit logging

**Cons**:
- Requires 200MB+ RAM minimum (500MB+ for HA)
- Requires seal/unseal management
- Adds operational complexity
- Different setup for local vs production

**Why not chosen**: Resource consumption and operational overhead conflict with zero-maintenance and commodity VPS constraints. The overhead would consume resources needed for actual applications.

### Option C: Docker Swarm Secrets

**Description**: Use Docker's built-in encrypted secrets storage.

**Pros**:
- Native Docker feature
- Encrypted at rest in Swarm

**Cons**:
- Requires Docker Swarm mode
- Not compatible with Docker Compose standalone

**Why not chosen**: Violates the "Docker Compose Only" constraint. Swarm mode introduces cluster management complexity.

---

## Consequences

### Positive

- Secrets not visible in `docker inspect`
- Zero RAM overhead for secret storage
- Same mechanism works in development and production
- File permissions provide host-level access control
- Idempotent generation - safe to run script multiple times

### Negative

- Applications without `_FILE` support require entrypoint wrapper scripts
- Secrets not encrypted at rest on host filesystem
- Rotation requires brief service restart (no zero-downtime rotation)

### Neutral

- Secrets directory must be created with correct permissions before first start

## Implementation Notes

### Directory Structure

```
secrets/
├── .gitkeep
├── postgres_password      # chmod 600
├── mysql_root_password    # chmod 600
├── mongodb_root_password  # chmod 600
└── ...
```

### Docker Compose Configuration

```yaml
secrets:
  postgres_password:
    file: ./secrets/postgres_password

services:
  postgres:
    secrets:
      - postgres_password
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password
```

### Generation Script Usage

```bash
# First-time setup
./scripts/generate-secrets.sh

# Safe to run again - only creates missing secrets
./scripts/generate-secrets.sh
```

### Application Compatibility

| Application | _FILE Support | Solution |
|-------------|---------------|----------|
| PostgreSQL | Yes | Use `POSTGRES_PASSWORD_FILE` |
| MySQL | Yes | Use `MYSQL_ROOT_PASSWORD_FILE` |
| MongoDB | Yes | Use `MONGO_INITDB_ROOT_PASSWORD_FILE` |
| Ghost | No | Entrypoint wrapper script |
| Generic | Varies | Test and document per-application |

### Exceptions: Applications Without _FILE Support

Some applications do not support the `_FILE` suffix convention and require the actual secret value in environment variables. Ghost is a notable example.

**Ghost Exception**:

Ghost requires database credentials directly in environment variables (`database__connection__password`). It does not support reading from `/run/secrets/`.

**Workaround**: Use an entrypoint wrapper script that reads the secret file and exports it as an environment variable before starting the application:

```dockerfile
# Custom entrypoint wrapper
#!/bin/sh
if [ -f /run/secrets/ghost_db_password ]; then
  export database__connection__password=$(cat /run/secrets/ghost_db_password)
fi
exec docker-entrypoint.sh "$@"
```

```yaml
# docker-compose.yml
services:
  ghost:
    image: ghost:5-alpine
    secrets:
      - ghost_db_password
    entrypoint: ["/custom-entrypoint.sh"]
    command: ["node", "current/index.js"]
```

This pattern maintains secret isolation (not in `docker inspect`) while accommodating applications that lack native `_FILE` support. The secret is only exposed in the container's environment after startup, not in the compose definition.
